Category 02 . RAG

📚 RAG：检索增强生成

RAG（Retrieval-Augmented Generation）检索增强生成，让 LLM 连接企业私有知识，给出精准回答

📄 文档解析与切片

  • 图片 + 视频 - 视觉模型 - 文本描述（保留原文件 URL，关联 URL + 描述）
  • 文本格式统一：PDF / Word / HTML / Markdown 统一处理
  • 智能切片：语义段落、重叠窗口、自适应分块

🔍 检索与召回

  • 向量检索（Embedding 语义匹配）+ 关键词检索（BM25）混合召回
  • 重排序（Reranker）：Cross-Encoder 精排 Top-K
  • 查询改写：问题扩展、多轮对话上下文理解

🧠 Prompt 注入与生成

  • 检索到的文本片段注入 System Prompt
  • LLM 基于源文档生成回答，附带引用来源
  • 支持多轮对话，记忆上下文

📊 评估与优化

  • RAG Assessment（RAGAS）评估框架：忠实度、相关性、精准度
  • A/B 测试不同切片策略与 Embedding 模型
  • 用户反馈闭环：点赞/踩 - 持续优化检索质量
RAG 是企业知识库的核心架构。完整流程：文档解析 - 智能切片 - 向量化 - 混合检索 - Prompt 注入 - LLM 生成。让 AI 基于企业真实数据给出可信的回答。

Category 02 . 重要因素

🔑 好的 RAG 要注意哪些因素？

构建高质量 RAG 系统的五个关键要素，直接决定回答质量

📄
文档质量 -- Garbage In, Garbage Out

扫描件 OCR 准确率、PDF 表格解析、图片中的文字提取。文档质量直接决定检索上限。

✂️
切片策略 -- 粒度决定精度

太小丢失上下文，太大检索不精准。推荐 chunk=512~1024 tokens，相邻 chunk 重叠 10%（即 chunk_size × 0.1，如 512→重叠51 tokens，1024→重叠102 tokens）。按文档类型自适应调整。

🔢
Embedding 模型 -- 语义理解的基石

text2vec-base-multilingual — 支持中英+多语言，轻量
paraphrase-multilingual-MiniLM-L12-v2 — 支持 50+ 语言，速度快
multilingual-e5-large — 微软出品，多语言效果好

🔍
检索策略 -- 混合召回 + 重排序

BM25（精确关键词） + Embedding（语义理解） 双路混合召回 → Cross-Encoder Reranker 对候选结果重新评分排序 → 输出 Top 3 最优片段给 LLM。

📝
Prompt 工程 -- 引导 LLM 正确使用检索结果

明确指令：基于提供的文档回答，不知道就说不知道。引用来源、禁止编造。RAG Assessment（RAGAS）框架持续评估。

RAG 质量五要素：文档质量 - 切片策略 - Embedding 模型 - 混合检索+重排序 - Prompt 工程。每个环节都可能成为瓶颈，需要系统性评估和持续优化。RAG Assessment（RAGAS）是最常用的评估框架。

Category 02 . 多模态

🖼️ 图片 / 视频 的 RAG 如何制作？

传统 RAG 只处理文字，多模态 RAG 让图片、视频也可检索和问答

📷
图片 → 三种理解路径

OCR（光学字符识别）：扫描件、证照等直接提取文字存储。
CLIP 向量嵌入：产品图、设计图生成向量，支持“以图搜图”“文字搜图”。
多模态 LLM：GPT-4o / Qwen-VL 深度理解图片内容，支持看图问答与自动标注。

🎬
视频 → 抽帧（图片）+ 语音转写（文本）+ 时间戳

每 2-3 秒抽一帧→走图片管线（OCR/CLIP/多模态 LLM）。
同时 Whisper 转写音频为文字。
时间戳将帧与文字绑定，检索时可精确跳转到视频对应秒数。

🔄
依赖大模型实际流程

图片/视频 → 多模态LLM理解 → 切片存储 → 用户提问 → 混合检索 → 多模态 LLM 生成回答 → 回答中引用图片/视频片段

多模态 RAG 扩展了传统 RAG 的能力边界。关键技术：OCR（光学字符识别）、CLIP多模态嵌入、Whisper语音转写、多模态LLM。适用场景：产品手册中的图表检索、培训视频知识库、设备故障照片诊断等。

Category 03 · 电商销量预测

电商SKU销量预测

💡 TFT · 多因素预测模型

① 根据：Google/Oxford 2021 论文，M5 时序预测竞赛前列，自带可解释性

② 输入

历史销量（店铺后台）+ 促销标签/折扣深度（商家后台活动配置）+ 城市天气（和风天气/OpenWeatherMap API）+ 节假日/大促日（人工维护大促日历表）+ 社媒搜索指数（Google Trends / 蝉妈妈 / 百度指数）

③ 工具

TFT 算法（Temporal Fusion Transformer）

⚡ 零代码：AWS Forecast | Google Vertex AI | 阿里云 PAI

④ 输出

未来30天日级销量预测表

📊 分层预测 · 量级分策略

① 根据：零售行业通行做法，不同量级商品统计特性不同，需分层建模

② 输入

商品日均销量分层（店铺后台订单数据）：爆款（>100件/天）、腰部（10-100）、长尾（<10）

③ 工具

爆款：DeepAR | 腰部：LightGBM | 长尾：Croston 间歇加权评分公式

⚡ 零代码：DeepAR → AWS Forecast  |  LightGBM → 阿里云 PAI / Azure AutoML  |  Croston → 需自写

均为专用时序预测算法，非语言大模型

④ 输出

爆款未来30天销量预测 / 腰部未来30天销量预测 / 长尾未来30天销量预测

⚡ TFT · 算力 & 成本

算力：高 — 全量 SKU 训练，需 GPU
成本：贵 — 按训练时长 + 调用次数计费
适合：SKU 少、数据丰富、预算充足

💰 分层 · 算力 & 成本

算力：低 — 仅爆款用 DeepAR，其余轻量
成本：省 — 总成本约 TFT 的 20-30%
适合：SKU 多、长尾占比大、控预算

17 / 27
左边TFT是单品精细预测，右边分层是全店策略。爆款用TFT日级精算，腰部用LightGBM周级够用，长尾用Croston判断会不会卖。好钢用在刀刃上——全店1000个SKU不用都上TFT。——它能同时处理销量、促销、天气、节假日等多种因素，输出未来30天的预测和置信区间。分层预测的关键：爆款要日粒度精细管理，长尾商品用间歇加权评分公式，不要一刀切。

Category 03 · 库存+调拨+预警

电商SKU - 安全库存 + 多仓调拨 + 滞销预警

📦 安全库存动态计算

① 输入

需求预测（上一页 TFT 输出），供应商历史交货天数，目标服务水平=98%

② 模型

Monte Carlo 仿真：每次随机抽取 TFT 某一天预测销量 + 供应商某一次历史交货天数，模拟一万次 → 计算不同库存水平下的断货概率
⚡ @RISK / Oracle Crystal Ball（零代码）

③ 输出

满足98% 不断货所需的最小安全库存量

🚚 多仓智能调拨

① 输入

各仓当前库存，各区域需求预测（← 上一页 TFT 输出），仓间运输成本

② 模型

整数规划（Google OR-Tools）求解最小值：运输成本 + 持有成本 + 缺货损失
⚡ Gurobi / CPLEX（企业版）

③ 输出

最优调拨方案：从哪仓调到哪仓、调多少

🚨 滞销预警

① 输入

Category 03 · 装备制造

⚡ 装备制造 — 库存管理 + 滞销预警

与电商不同，实业库存面向 BOM 物料、备件、在制品，由生产排期和设备可靠性驱动

📦 BOM 物料需求预测

① 输入

生产订单交付日期 + BOM 物料清单（铜/硅钢片/绝缘材料用量）+ 供应商历史交货天数 + 原料价格趋势

② 模型

MRP 反向推导 + 时序预测：订单交期 → 开工日期 → 原料到货日期，结合供应商交货波动算安全库存

③ 输出

每批次订单的原料采购计划表 + 安全库存量

🔧 备件库存管理

① 输入

在役设备分布地图 + 各型号故障率曲线 + 备件历史消耗 + 采购周期

② 模型

泊松分布 / Weibull 故障模型 → 按区域仓库计算备件需求概率，保证 95% 不断供

③ 输出

每个区域仓库建议备件清单 + 补货触发点

🚨 滞销/呆湞预警

① 输入

原料库龄 + 在制品湞留天数 + 成品库存周转率 + 订单取消/变更记录

② 模型

规则引擎：库龄(30/60/90天) + 订单状态(是否已排产) + 替代通用性(专用/通用料) → 三级预警

③ 输出

呆湞物料清单 + 建议处理方案（调拨/折价/报废）

装备制造库存与电商库存的本质区别：电商看历史销量曲线，实业看生产排期和设备可靠性。三个核心场景：BOM物料需求预测、备件库存、呆湞预警。
